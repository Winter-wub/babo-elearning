## Authentication Flow

### Credentials and JWT Strategy

The platform uses **Auth.js v5** (NextAuth) with a credentials provider and JWT session strategy. There is no database-backed session table -- sessions are entirely stateless.

**Login flow:**

1. User submits email and password on the login page.
2. The credentials provider in `auth.ts` looks up the user by email.
3. If the user exists and `isActive` is `true`, the password is compared against the stored bcrypt hash (12 salt rounds).
4. On success, a JWT is issued containing the user's `id` and `role` claims.
5. The JWT is stored as an HTTP-only cookie with a 30-day max age.
6. Subsequent requests include the cookie automatically. Auth.js validates the JWT on each request.

**Registration flow:**

1. User submits name, email, and password via the registration form.
2. Input is validated with Zod (email format, password 8-128 chars, name 2-100 chars).
3. The server checks for duplicate email addresses.
4. Password is hashed with bcrypt (12 rounds) and stored.
5. A new user record is created with the STUDENT role.

**Key details:**

| Property | Value |
|---|---|
| Password hashing | bcryptjs, 12 salt rounds |
| Session strategy | JWT (stateless) |
| Session max age | 30 days |
| Token claims | `id`, `role` |
| Edge compatibility | Auth config split into `auth.config.ts` (edge-safe) and `auth.ts` (Node-only) |

## Authorization Layers

The platform enforces authorization at four layers. A request must pass all applicable layers to succeed.

### Layer 1: Middleware (Route Protection)

The Auth.js `authorized` callback in `auth.config.ts` runs on every request via Next.js middleware. It enforces route-level access rules:

| Route Pattern | Rule |
|---|---|
| `/admin/*` | Must be authenticated with ADMIN role. Students are redirected to `/dashboard`. Unauthenticated users go to `/login`. |
| `/dashboard`, `/videos/*` | Must be authenticated (any role). Unauthenticated users go to `/login`. |
| `/login`, `/register` | Authenticated users are redirected to their dashboard (admin or student). |
| Everything else | Allowed (public pages, static assets). |

### Layer 2: Page-Level Checks

Server components fetch data using server actions. If the action returns an error (unauthorized, not found), the page renders an appropriate error state or redirects.

### Layer 3: Server Actions

Every server action verifies the session and role before executing business logic:

- `requireAdmin()` -- checks for a valid session with `role === "ADMIN"`. Throws `"Unauthorized"` otherwise.
- `requireAuth()` -- checks for any valid session. Throws `"Unauthorized"` otherwise.

Additional authorization logic within actions:
- `getVideoById` checks that a STUDENT has an explicit `VideoPermission` record.
- `getPermittedVideos` returns only videos the current user has permission for.
- `deleteUser` and `updateUser` prevent admins from deactivating their own account.

### Layer 4: API Route Handlers

The `/api/videos/[videoId]/signed-url` route handler performs its own authentication and authorization independently of middleware:

1. Validates the session exists.
2. Validates the Origin/Referer header.
3. Verifies the video exists and is active.
4. Checks that the student has a `VideoPermission` record (admin bypasses this check).

## Video Protection Measures

### Signed URLs (Primary Protection)

The R2 bucket is **private** -- there is no public read access. All video playback goes through time-limited signed URLs.

| Property | Value |
|---|---|
| Signing method | `@aws-sdk/s3-request-presigner` with `GetObjectCommand` |
| Expiry | 15 minutes (900 seconds) |
| Generated by | `/api/videos/[videoId]/signed-url` route handler |

The client-side video player fetches a signed URL and refreshes it before expiry. Each URL generation requires passing all authorization checks.

### Origin and Referer Validation

The signed URL endpoint validates that the request originates from the platform:

- The `Origin` header must exactly match `NEXT_PUBLIC_APP_URL`, **or**
- The `Referer` header must start with `NEXT_PUBLIC_APP_URL`.

Requests without either header are rejected with a 403 status. This prevents other websites from embedding the platform's video player or calling the signed URL endpoint from their pages.

### CORS Lockdown

The `next.config.ts` sets restrictive CORS headers on all `/api/videos/*` routes:

```
Access-Control-Allow-Origin: <NEXT_PUBLIC_APP_URL>
Access-Control-Allow-Methods: GET, OPTIONS
Access-Control-Allow-Headers: Content-Type, Authorization
```

Additionally, the R2 bucket itself should have a CORS policy restricted to the platform domain (see [deployment guide](deployment.md)).

### Content Security Policy

The following CSP directives protect against content injection:

| Directive | Value | Purpose |
|---|---|---|
| `default-src` | `'self'` | Block all external resources by default |
| `media-src` | `'self' blob:` | Allow video from same origin and blob URLs |
| `connect-src` | `'self' https://*.r2.cloudflarestorage.com` | Allow API calls and R2 signed URL fetches |
| `frame-ancestors` | `'self'` | Prevent iframe embedding on other sites |
| `object-src` | `'none'` | Block Flash/plugin embeds |

Additional response headers:

| Header | Value |
|---|---|
| `X-Frame-Options` | `SAMEORIGIN` |
| `X-Content-Type-Options` | `nosniff` |
| `Referrer-Policy` | `strict-origin-when-cross-origin` |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` |

### No-Cache on Signed URLs

Every signed URL response includes:

```
Cache-Control: no-store, no-cache, must-revalidate
Pragma: no-cache
```

This ensures browsers and proxies do not cache signed URLs, which would allow them to be reused after expiry or access revocation.

## Client-Side Deterrents

These measures raise the effort required for casual content theft. They are deterrents, not security barriers.

| Measure | Description |
|---|---|
| Right-click disabled | The `contextmenu` event is suppressed on the video container. |
| No download button | Vidstack player is configured without a download button. |
| No `<a download>` | No download links are rendered anywhere in the video UI. |
| CSS `user-select: none` | Text selection is disabled on the player area. |

## Known Limitations

1. **No DRM.** The platform uses MP4 files with signed URLs. This is an MVP approach. A determined user with browser developer tools can extract the signed URL and download the video file directly. True content protection requires DRM (Widevine, FairPlay) which adds significant complexity.

2. **Screen recording is not preventable.** No web-based solution can fully prevent screen recording via OS-level tools or external capture devices.

3. **Signed URLs are bearer tokens.** Anyone who obtains a valid signed URL can access the video from any network location until the URL expires (15 minutes). Sharing the URL within the expiry window grants access.

4. **No HLS segmentation.** Videos are served as single MP4 files, not segmented HLS/DASH streams. A single signed URL grants access to the entire file.

5. **No rate limiting on signed URL generation.** The `/api/videos/[videoId]/signed-url` endpoint does not currently limit request volume. A compromised session could generate many URLs rapidly.

6. **CSP includes `unsafe-eval`.** The current CSP allows `unsafe-eval` in the `script-src` directive, which is required by the Next.js development server but should be removed in production.

## Recommendations for Production Hardening

The following improvements are recommended before deploying to production with sensitive content.

### Rate Limiting

Implement rate limiting on the signed URL endpoint to prevent URL generation abuse.

**Recommended approach:** Use Upstash Ratelimit with Redis or Vercel KV.

```typescript
// Example: 10 requests per minute per user
const ratelimit = new Ratelimit({
  redis: Redis.fromEnv(),
  limiter: Ratelimit.slidingWindow(10, "1 m"),
});

const { success } = await ratelimit.limit(`signed-url:${session.user.id}`);
if (!success) {
  return NextResponse.json({ error: "Too many requests" }, { status: 429 });
}
```

### Account Lockout

Add temporary account lockout after repeated failed login attempts (e.g., 5 failures in 15 minutes triggers a 30-minute lockout). This prevents brute-force password attacks.

### Password Complexity

Enforce stronger password requirements beyond the current 8-character minimum:
- Require at least one uppercase letter, one lowercase letter, one digit, and one special character.
- Check against common password lists.
- Enforce maximum password age with rotation reminders.

### CSP Hardening

Remove `'unsafe-eval'` from the `script-src` directive in production builds. Use nonce-based CSP instead:

```typescript
const cspDirective = process.env.NODE_ENV === "development"
  ? "script-src 'self' 'unsafe-eval' 'unsafe-inline'"
  : "script-src 'self' 'unsafe-inline'";
```

### Additional Recommendations

| Area | Recommendation |
|---|---|
| **HTTPS** | Enforce HTTPS in production. Set `Strict-Transport-Security` header. |
| **Audit logging** | Log admin actions (user creation, permission changes, video uploads) with timestamps and actor IDs. |
| **Session revocation** | Consider a server-side session store (Redis) to enable immediate session invalidation when accounts are deactivated. |
| **Upload validation** | Validate uploaded files server-side (check magic bytes, scan for malware). |
| **DRM** | For high-value content, migrate to Cloudflare Stream with signed tokens or implement HLS with per-segment signing. |
| **Bulk operation limits** | Add array size validation to `bulkGrantPermissions` and `bulkRevokePermissions` to prevent abuse (e.g., max 100 items). |
| **CORS on R2** | Ensure the R2 bucket CORS policy only allows the platform origin (see [deployment guide](deployment.md)). |
| **Dependency scanning** | Run `pnpm audit` regularly and integrate Dependabot or Snyk for automated vulnerability alerts. |
